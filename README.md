# kubernates-schedular

The Kubernetes scheduler is a core control-plane component responsible for deciding where Pods run inside a cluster. When a Pod is created, it does not immediately start on a node; instead, it enters a Pending state, and the scheduler evaluates all available nodes to determine the most suitable one. This is necessary because clusters typically consist of multiple nodes with different capacities (CPU, memory, GPU) and purposes (production, development, specialized workloads). Without a scheduler, Pods could be unevenly distributed, resources could be wasted, and applications might fail due to overload or improper placement. The scheduler ensures efficient resource utilization, load balancing, and high availability by intelligently assigning Pods to nodes.

Internally, the scheduler works in stages. First is the filtering phase, where it removes nodes that do not meet the Pod’s requirements, such as insufficient resources, mismatched labels, or restrictions due to taints. Next is the scoring phase, where the remaining nodes are ranked based on factors like resource availability, balance, and preference rules. Finally, in the binding phase, the Pod is assigned to the highest-scoring node. This process happens automatically for every Pod unless explicitly overridden.

In real-world usage, we control scheduling behavior rather than replacing it. The simplest method is using nodeSelector, where Pods are assigned to nodes with specific labels. For more flexibility, node affinity allows defining advanced rules such as “must run” or “prefer to run” on certain nodes. Taints and tolerations work in the opposite direction by restricting nodes so that only specific Pods can run on them, which is useful for dedicated workloads like GPUs or databases. Additionally, resource requests (CPU and memory) play a critical role, as the scheduler uses them to ensure a node has enough capacity before placing a Pod.

In cases where default scheduling logic is not sufficient, Kubernetes allows the use of a custom scheduler. This involves running another scheduler alongside the default one and configuring Pods to use it via the schedulerName field. A custom scheduler can implement organization-specific logic, such as cost optimization, priority handling, or specialized hardware awareness. However, in most DevOps scenarios, engineers rely on the default scheduler and control placement using labels, affinity rules, and resource configurations rather than building a scheduler from scratch.
